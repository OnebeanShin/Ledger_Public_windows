# 월간 재정 리포트 생성 지시서 (Monthly Financial Report Playbook)

> 이 문서는 **AI 에이전트가 매월 말, 이 hledger 가계부 데이터로 전문 월간 재정 리포트(한국어·HTML)를
> kami 디자인 시스템으로 일관되게 생성**하도록 규정하는 지시서다.
> 리포트를 만들 때마다 이 문서를 처음부터 끝까지 따른다.

---

## 0. 역할 · 원칙

- 역할: **개인 재무 분석가 / 회계사**.
- 모든 수치는 hledger 원천 데이터에서 추출한 **사실**이어야 한다. 추정·보간 시 반드시 명시한다.
- 조언·예측이 아니라 **관찰과 사실**을 기술한다 (예: "식비가 전월 대비 12% 증가" O / "식비를 줄이세요" 지양).
- **검증 없이 완료 처리하지 않는다** (§7 체크리스트 필수).
- 기준 통화 `KRW`, 시간대 `Asia/Seoul`, 시작 파일 `main.journal`.

---

## 1. 실행 트리거 · 입력

- 트리거 예: "2026년 5월 재정 리포트 만들어줘", "지난달 결산 리포트".
- 입력 파라미터:
  - 대상 월 `YYYY-MM` (예: `2026-05`). 미지정 시 **직전 종료 월**을 기본값으로 하고 사용자에게 한 줄 확인.
- 저장소 루트: 이 지시서가 있는 `webui/report/plan/` 기준 세 단계 위의 폴더.
- 배포판에서도 동일하게 동작한다. 에이전트는 사용자가 결산 리포트 생성을 요청하면 이 파일을 먼저 읽고, 결과 HTML을 반드시
  `webui/report/<YYYY-MM>/<YYYY-MM>-financial-report.html`에 저장한다. WebUI 결산 탭은 `/api/reports`가 이 경로를 스캔하므로,
  파일이 생성되면 새로고침 후 바로 목록과 월 이동 컨트롤에 반영된다.

## 2. 선행 준비

리포트 생성 전 항상 시세를 최신화한다.

```bash
cd "<저장소 루트>"
./scripts/update-prices.sh        # macOS/Linux
python scripts/update-prices.py   # Windows 배포판
```

둘 중 현재 배포판에 존재하는 스크립트만 실행한다. 가격 갱신이 실패하면 실패 원인과 사용한 가격 기준일을 리포트 감사 섹션에 남긴다.

월 경계 변수(이후 명령에서 사용):

```bash
M0=2026-05-01          # 대상 월 1일
M1=2026-06-01          # 다음 달 1일 (hledger -e 는 배타적 종료)
PREV0=2026-04-01       # 전월 1일 (MoM 비교용)
Y12=2025-06-01         # 12개월 추이 시작
```

## 3. 데이터 추출 — hledger 명령 (검증됨, hledger 1.52)

모든 명령은 저장소 루트에서 `-f main.journal` 로 실행한다.

| 목적 | 명령 |
|---|---|
| 손익(월) | `hledger -f main.journal incomestatement -b $M0 -e $M1` |
| 현금흐름(월) | `hledger -f main.journal cashflow -b $M0 -e $M1` |
| 지출 카테고리(트리) | `hledger -f main.journal balance expenses --tree --depth 2 -b $M0 -e $M1` |
| 지출 상세(말단) | `hledger -f main.journal balance expenses -b $M0 -e $M1` |
| 재무상태표(월말, 평가) | `hledger -f main.journal balancesheet -e $M1 --value=now,KRW --depth 3` |
| 순자산 추이(12개월) | `hledger -f main.journal balancesheet -M -b $Y12 -e $M1 --value=now,KRW --depth 1` |
| 투자 평가액 | `hledger -f main.journal balance assets:investment --value=now,KRW` |
| 투자 취득원가·평가손익 | `python3 scripts/investment-gain.py` (매수일 환율 기반 KRW 원가) |
| 부채 잔액 | `hledger -f main.journal balance liabilities --depth 2 -e $M1` |
| 전월 손익(MoM) | `hledger -f main.journal incomestatement -b $PREV0 -e $M0` |
| 복식부기 검증 | `hledger -f main.journal check` |

> **현금흐름 주의**: `cashflow`는 현금성 계정(`assets:bank:입출금`)만 반영한다. 투자 매수/매도는 포함되지 않으므로
> 리포트에는 "은행 현금 기준"임을 명시한다.

## 4. 지표 계산 규칙 (승인된 정의)

- **총수입** = income 전체 합 (월급·용돈·선물·이자·중고거래·기타 포함).
- **총지출** = expenses 전체 합.
- **순저축** = 총수입 − 총지출 (= incomestatement 의 Net).
- **저축률** = 순저축 ÷ 총수입 × 100 (소수 1자리 %). 비정기 수입(용돈·선물 등)도 분모에 포함.
- **순자산** = 자산(평가액) − 부채 (= balancesheet Net, `--value=now,KRW`).
- **MoM 변화** = 당월 값 − 전월 값 (금액 및 %).
- **투자 평가손익(미실현)** = 현재 평가액 − 취득원가. **수입에 포함하지 않고 "평가손익"으로 별도 표기**한다.
  - **취득원가(KRW)** = Σ 각 매수 로트(수량 × USD 단가 `{USD …}` × 매수일 환율).
  - **환율 규칙**: prices.journal 의 USD/KRW 사용 — 매수일에 환율이 있으면 그 값을, 없으면 **날짜상 가장 가까운** 환율을 사용.
  - 단기매매 무상 취득분(`{USD 0}`)은 원가 0. 산출 도구: `scripts/investment-gain.py`.
- **금액 표기**: hledger 원시 출력은 소수점이 길다(예: `99901.000…`). 리포트에서는 **KRW 0자리 반올림 +
  천단위 콤마**(예: `99,901원`)로 가공한다. 비율은 소수 1자리.

## 5. 리포트 구조 (11개 섹션)

kami `long-doc-ko` 템플릿 위에 아래 순서로 구성한다. 각 섹션의 데이터원·차트는 다음과 같다.

1. **표지/헤더** — 제목 "YYYY년 M월 재정 리포트", 기간, 생성일, 기준통화(KRW). (template cover 영역)
2. **Executive Summary** — KPI 타일 6개: 총수입 · 총지출 · 순저축 · 저축률 · 월말 순자산 · 순자산 MoM 변화. 3줄 핵심 요약.
3. **손익 분석** — incomestatement 표 + **Waterfall**(수입 → 지출 → 순저축).
4. **지출 분석** — 카테고리별 **Donut**(상위 비중) + 상위 카테고리 **Bar**(전월 대비 증감). 데이트/식비/치대 등 대분류 기준.
5. **현금흐름** — 은행 현금 변동 요약(은행 현금 기준 명시).
6. **부채** — 학자금(`liabilities:loan:학자금`) 잔액 · 상환 진행.
7. **자산 · 순자산** — balancesheet 표 + 순자산 12개월 **Line chart**.
8. **투자 포트폴리오** — BTC·주식 평가액 / **평가손익(미실현, 별도)**, 자산 비중 **Donut**, (옵션) BTC **Candlestick**.
9. **투자 테마 동향(보유 투자자산 기반)** — §5.2 가이드. 보유 종목과 상품에 맞춘 이번 달 주요 사건 + 다음 달 예정 일정(웹 검색).
10. **결론** — §5.1 가이드(정성적 분석)에 따라 6개 소제목(h3)으로 작성.
11. **부록 · 감사(Audit)** — `hledger check` 결과, 리포트 합계와 원천 수치 일치 확인표. (별도 파일 아님, 본 리포트 마지막 섹션)

## 5.1 정성적 분석 작성 가이드 (전문 가이드라인)

리포트의 "정성적 분석"은 숫자 나열이 아니라 **수치를 해석**하는 서술이어야 한다.
아래 6개 소제목(h3) 구조로 작성하며, **모든 정성 주장에는 근거 수치를 병기**한다.

1. **종합 진단** — 흑/적자·저축률 추세·수입의 질·순자산 방향을 2~3문장 요약.
2. **수입·지출 구조 · 수입의 질** — 정기(근로) vs 비정기(용돈·선물·중고 등) 수입 비중, 흑자가 계속 이어질 수 있는지, 비정기 수입을 뺀 평소 흑자(기본 흑자).
3. **지출 행태** — MoM 증감 드라이버를 항목별로 짚고 **일시성 vs 구조적** 지출을 구분. 최대 비중 카테고리 해석.
4. **자산 · 포트폴리오 건전성** — 자산 집중도(%), 단일 종목 비중, 미실현 손익 해석, 유동성 = 현금 ÷ 월지출(개월수).
5. **리스크 관찰** — 수입 변동성 / 자산 집중·변동성 / 유동성 / 부채 등 4개 내외, 각 1줄 + 수치.
6. **다음 달 관찰 포인트** — 일시성 확인, 평소(비정기 수입 제외) 저축력, 현금 여유(비상금) 추이 등 다음 달 확인 항목.

작성 원칙:
- **사실·관찰 중심, 조언 지양**: "줄이세요" 같은 처방 대신 "일시성 확인이 필요하다"처럼 관찰·점검 표현 사용.
- 모든 정성 문장에 수치 근거(원·%·배수·개월수) 동반. 막연한 형용사("많다/적다")만 쓰지 않는다.
- 예측·투자 권유 금지. 추세는 "둔화/가속" 등 관측된 변화로만 기술.
- **쉬운 우리말 우선**: 뜻이 모호한 외래어·한자어를 피하고 누구나 바로 이해할 표현을 쓴다. 아래 용어는 쓰지 않는다.

  | 쓰지 말 것 | 대신 쓸 표현 |
  |---|---|
  | buffer · 완충 | 현금 여유(비상금), 몇 개월치 생활비 |
  | 베이스라인 · baseline | 평소 기준 · 기본 (예: 기본 흑자) |
  | 구조적 수지 | 평소 흑자·적자(꾸준한 수입·지출만 따진 것) |
  | 수지 여력 | 쓰고 남는 여윳돈 · 저축 여유 |
  | 수지 | 수입·지출(흑자/적자) |

- 권장 파생 지표: 저축률, MoM %, 자산 집중도(%), 미실현 수익률(%), 유동성 개월수(현금÷월지출), 순자산 증가율.

## 5.2 투자 테마 동향 작성 가이드 (보유 투자자산 기반 웹 검색)

투자 테마 동향은 고정 주제가 아니라 **사용자의 실제 보유 투자자산**에 맞춰 매월 유동적으로 정한다.

작성 절차:
1. 먼저 보유 투자자산을 확인한다.
   - 우선 `python3 scripts/investment-gain.py`와 `hledger -f main.journal balance assets:investment --value=now,KRW` 결과를 본다.
   - 보조로 `webui`가 실행 가능한 환경이면 `/api/portfolio` 응답의 `holdings`를 확인한다.
   - `assets:investment:btc`는 `BTC`, `assets:investment:stock:<SYMBOL>`은 `<SYMBOL>`을 주된 조사 단위로 본다.
2. 보유 자산을 관련 테마로 묶는다. 예:
   - `BTC` 또는 비트코인 관련 종목: 비트코인, 현물 ETF, 채굴, 규제, 금리·달러 유동성.
   - `MSTR`: 비트코인 트레저리 기업, 레버리지성 비트코인 노출, 전환사채·자본조달.
   - `MSFT`: 클라우드, AI 인프라, Copilot, Azure, 엔터프라이즈 소프트웨어.
   - `GOOGL`: 검색·광고, Gemini, 클라우드, AI 모델·칩, 반독점·규제.
   - 새 심볼이 추가되면 회사명, 사업 부문, 주요 매출원, 투자 노출을 검색해 테마를 새로 만든다.
3. 웹 검색으로 각 테마별 **이번 달 주요 사건**과 **다음 달 예정·기대 일정**을 수집한다.
   - 회사 보유 종목은 IR, 실적 발표, 제품 발표, 규제 이슈, 주요 컨퍼런스, 애널리스트가 아닌 1차 출처와 주요 언론을 우선한다.
   - 상품·거시 테마는 ETF 자금흐름, 정책·규제, 금리·환율, 주요 네트워크 지표 등 자산과 직접 관련된 사건을 우선한다.
4. 섹션 제목은 보유자산에 맞춰 만든다. 고정으로 "비트코인 스탠다드 / AI 특이점"을 쓰지 않는다.
   - 예: `비트코인 · MSTR`, `AI 인프라 · MSFT/GOOGL`, `클라우드 소프트웨어`, `반도체·AI 가속기`.
5. 원칙: 날짜·사실 명시, 투자 권유·단정적 예측 금지, 미래 일정은 "예정/기대"로 표기(변동 가능), **출처 도메인 명기**.
6. 반드시 **보고서 시점 기준 검색**을 수행한다. 기억(학습데이터)에 의존하지 않는다.
7. 보유 투자자산이 없으면 이 섹션 제목을 `투자 테마 동향 · 해당 없음`으로 쓰고, "월말 기준 보유 투자자산이 없어 별도 시장 동향 조사를 생략했다"고 명시한다.

## 6. kami 디자인 적용

kami 스킬을 사용한다.
스킬이 없으면 https://github.com/tw93/kami 에서 다운받아서 설치한다.

1. **베이스 템플릿 복사**: `assets/templates/long-doc-ko.html` 를 출력 위치로 복사 후 `{{placeholder}}` 를 채운다.
   - 채울 주요 placeholder: `{{문서 제목}}`, `{{작성자}}`, `{{요약}}`, `{{키워드}}`, cover/eyebrow/title/sub, 각 챕터 본문·표·callout.
2. **디자인 토큰** (`references/design.md`, `references/tokens.json` 준수):
   - 캔버스 parchment `#f5f4ed` (순백 금지), accent ink-blue `#1B365D` **단일**.
   - 중성색은 따뜻한 톤만(쿨 그레이 금지), serif 위계(본문 400 / 제목 500), 하드 섀도 금지.
   - 본문 텍스트는 진한 검정(`#141414`)으로 가독성 확보(흐린 회갈색 지양).
   - 태그/배지 배경은 **solid hex만** (rgba 금지 — WeasyPrint double-rectangle 버그).
3. **차트 임베드** (`references/diagrams.md` 준수):
   - 사용 차트 파일: `assets/diagrams/waterfall.html`, `donut-chart.html`, `bar-chart.html`,
     `line-chart.html`, `candlestick.html`.
   - 각 파일의 `<svg>...</svg>` 블록을 복사해 long-doc 의 `<figure>` 안에 붙이고 데이터 값을 치환한다.
   - 색은 ink-blue accent + 중성 ramp 1개(2 ramp 이내). diagrams.md 의 anti-slop 표를 지킨다.
4. **폰트**: 한국어 → Source Han Serif K (kami 번들, `assets/fonts/SourceHanSerifKR-*.otf`). 페이지당 단일 serif.
5. 막히면 kami `SKILL.md` 와 `CHEATSHEET.md` 를 읽고 규칙을 따른다.

## 7. 출력 · 네이밍

- 출력 폴더: `webui/report/<YYYY-MM>/` (ISO, 예 `webui/report/2026-05/`).
- 파일명: `<YYYY-MM>-financial-report.html` (감사 섹션 포함, **단일 파일**).
- 산출 형식: **단독 HTML**(브라우저에서 바로 열림, 폰트·스타일 인라인/상대경로).
- PDF가 필요하면 WeasyPrint 경유(`python3` + weasyprint). 현재 미설치 — 필요 시 설치 안내만 하고 HTML은 항상 제공.
